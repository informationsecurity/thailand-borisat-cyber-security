## คำถามที่พบบ่อย (FAQ)

### Penetration Test คืออะไร?
เป็นการจำลองการโจมตีแบบควบคุมที่ใช้เพื่อตรวจสอบว่าระบบ, แอปพลิเคชัน, API, เครือข่าย หรือ cloud สามารถถูกใช้ประโยชน์โดยผู้โจมตีจริงได้หรือไม่ แตกต่างจากการสแกนธรรมดา Penetration Test ผสมการวิเคราะห์ทางเทคนิค การตรวจสอบแบบ manual และการจัดลำดับความเสี่ยง

### เมื่อใดและเหตุใดจึงจำเป็น?
มีประโยชน์เมื่อคุณต้องเปิดตัวบริการใหม่, เปลี่ยน infrastructure, เผชิญกับ audit หรือตรวจสอบความปลอดภัยหลังการเปลี่ยนแปลงที่สำคัญ ในปี 2026 มีความสำคัญเชิงกลยุทธ์มากขึ้นเพราะหลายองค์กรมี attack surface ที่กว้างขึ้น, API มากขึ้น, cloud มากขึ้น และ integration ภายนอกมากขึ้น

### ค่าใช้จ่ายเฉลี่ยของ Penetration Test เท่าไร?
ขึ้นอยู่กับ scope, ความซับซ้อน, จำนวน asset, ความลึกที่ต้องการ และประเภทของการทดสอบ การทดสอบบนแอปพลิเคชันเดียวอาจมีค่าใช้จ่ายต่างกันมากเมื่อเทียบกับการประเมินแบบ multi-domain บน cloud, hybrid infrastructure หรือสภาพแวดล้อมที่มีการกำกับดูแล

### จะเลือกผู้ให้บริการที่เหมาะสมได้อย่างไร?
ดูที่ห้าองค์ประกอบ: ความสามารถทางเทคนิค, วิธีการ, คุณภาพของรายงาน, การสนับสนุน remediation และความสามารถในการเข้าใจบริบทธุรกิจของคุณ หากผู้ให้บริการสามารถอธิบาย scope, ความเสี่ยง และลำดับความสำคัญได้ดี นั่นเป็นสัญญาณที่ดีของความเป็นผู้เชี่ยวชาญ

### การรับรองที่สำคัญมีอะไรบ้าง?
ในบรรดาที่ได้รับการยอมรับมากที่สุดได้แก่ OSCP, CEH, CISSP, GPEN และ GWAPT ไม่เพียงพอด้วยตัวเอง แต่ช่วยแยกแยะทีมที่มีการฝึกอบรมแข็งแกร่งจากทีมทั่วไป สำหรับโครงการที่มีการกำกับดูแล ประสบการณ์ในอุตสาหกรรมและความคุ้นเคยกับมาตรฐาน เช่น PCI DSS, GDPR, NIS2 และ DORA ก็มีความสำคัญเช่นกัน

### Black box, white box หรือ grey box ดีกว่ากัน?
ขึ้นอยู่กับวัตถุประสงค์ของคุณ Black box จำลองผู้โจมตีภายนอกที่มีข้อมูลน้อย; white box เหมาะกับการตรวจสอบเชิงลึกและรวดเร็ว; grey box สมดุลระหว่างความสมจริงและความลึก การเลือกที่ถูกต้องไม่ใช่ที่ "แข็งแกร่งที่สุด" แต่เป็นสิ่งที่มีประโยชน์มากที่สุดสำหรับความเสี่ยงจริงของคุณ

### Penetration Test เพียงอย่างเดียวเพียงพอหรือไม่?
ไม่ มันเป็น snapshot ที่มีประโยชน์มากแต่ไม่สามารถแทนโปรแกรม vulnerability management ที่ต่อเนื่องได้ คุณค่าเพิ่มขึ้นเมื่อการทดสอบตามด้วย remediation, retest และการติดตามเป็นระยะ

### รายงานควรละเอียดแค่ไหน?
มากพอที่จะช่วยให้ทีมเทคนิคแก้ไขและผู้บริหารเข้าใจผลกระทบ รายงานที่ดีรวมถึงบริบท, หลักฐาน, ความเสี่ยง, ลำดับความสำคัญ, remediation และหากเป็นไปได้ การสรุปเชิงผู้บริหารที่ชัดเจน

### การรับรองสำคัญกว่าประสบการณ์หรือไม่?
ทั้งสองอย่างมีความสำคัญ การรับรองแสดงความเข้มงวดและการเตรียมพร้อม; ประสบการณ์แสดงให้เห็นว่าทักษะเหล่านั้นถูกนำไปใช้ในสภาพแวดล้อมจริง ด้วยข้อจำกัดด้านเวลา, compliance และธุรกิจ

### เหตุใดผู้ให้บริการบางรายจึงเหมาะกับ PMI และบางรายเหมาะกับ enterprise?
เพราะแนวทาง, กระบวนการ, ความลึก และรูปแบบการมีส่วนร่วมแตกต่างกัน PMI ต้องการความชัดเจน, ความรวดเร็ว และความเรียบง่ายในการดำเนินงาน; enterprise อาจต้องการ governance, การบูรณาการหลายทีม และการทดสอบในสภาพแวดล้อมที่กว้างมาก
